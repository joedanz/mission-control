# `mc` — Mission Control CLI (for agents)

`mc` is a global CLI for reading/writing this app's projects + tasks directly against Neon.
Install once: `npm link` (from this repo). Full reference: [`cli/README.md`](cli/README.md).

**Credentials** (resolved in this order): `AGENT_DATABASE_URL` env var → `$MC_ENV_FILE` →
`~/.config/mc/env` (must be `chmod 600`). The CLI refuses to run without the scoped
`AGENT_DATABASE_URL` (no silent fallback to `DATABASE_URL` — opt in with
`MC_ALLOW_DATABASE_URL_FALLBACK=1`). **The CLI has NO sign-in — anyone with the credential writes
with no email allowlist.** It is scoped at the DB layer to `projects` + `tasks` + `runs` + `events`
(read/write; no DELETE on runs/events; no auth tables).

**Output:** always pass `--json` (or pipe — JSON is the default when stdout is not a TTY).
Envelope: `{"ok":true,"command":"...","data":...}` or
`{"ok":false,"command":"...","error":{"code","message","field?"}}`.
Lists return `data:{items,count}`; mutations return the affected row (so you get the new `id`).
Exit codes: `0` ok · `1` DB/conflict · `2` validation · `3` not-found · `4` config/credentials.

**Discover the surface at runtime:** `mc spec --json` (every command, with `readonly` tags) and
`mc enums --json` (valid values for category/status/accent/priority/taskStatus/integration*).

**Addressing:** projects are addressed by **slug**, tasks by **id** (uuid). On VALIDATION errors
the message lists the valid values; on NOT_FOUND it suggests `mc project list`.

```
mc project list [--category --status --archived active|archived|all --search --limit]
mc project get <slug>
mc project add --name <n> --category <c> [--status --accent --domain --tech --repo-path --repo-url --live-url --priority --notes --sentry-project --email-provider --email-address --stripe-site]
mc project update <slug> [any add flag; only provided flags change]
mc project rm <slug> --yes        # cascades tasks; refuses without --yes
mc project set-repo <slug> <path> [url]
mc task list <slug> [--status --kind custom|integration]
mc task get <id>
mc task add <slug> <label...>
mc task set-status <id> <status>  # idempotent — prefer over toggle
mc task move <id> [--status <s>] [--top|--after <id>]  # board move: change column &/or reorder; --top = claimed next; refuses live-claimed → CONFLICT
mc task toggle <id>
mc task rm <id> --yes
mc task next [--project <slug>]                 # next claimable task; board order (sort_order) then FIFO; unclaimed/expired
mc task claim <id> [--run <id>] [--ttl <secs>]  # claim a task for the current run (race-safe; loses → CONFLICT)
mc task import-issues <slug> [--state open|closed|all] [--label <n>] [--limit <n>] [--dry-run]  # GitHub issues → tasks (idempotent by issue #; needs repoUrl + gh CLI)
mc integration set <slug> <type> <status>   # upsert; idempotent
mc integration list <slug>
mc composio catalog                          # list supported Composio toolkits (slug, name, tool count); no DB needed; reads COMPOSIO_API_KEY from host env (never stored)
mc composio connect <slug> <toolkit>         # start a connection; prints OAuth authorize URL; follow up with mc composio status. At most one connection per (project, toolkit)
mc composio status <slug> <toolkit>          # poll Composio and persist connection status (initializing|active|error|expired|disconnected)
mc composio list <slug>                      # list a project's Composio connections and their statuses
mc composio disconnect <slug> <toolkit>      # revoke at Composio and mark disconnected locally
mc composio mcp-config <slug>                # resolve active connections into an mcpServers JSON map (daemons call this at spawn to auto-feed the agent with remote-http MCP servers for Linear/Slack/…)
mc composio refresh <slug>                   # re-poll all of a project's connections; emit composio.connection_changed events on status changes
mc workflow list [--project <slug>]          # list agentic workflows (node graphs of agent runs + integrations)
mc workflow get <slug>                       # one workflow (graph + status + version)
mc workflow create --project <slug> --name <n> [--slug <s>] [--graph <json|@file>] [--description <d>]   # create a workflow from a React Flow {nodes,edges} graph. A provided graph is validated (DAG, exactly one trigger node, agent nodes need a prompt; every {{nodeId.field}} ref must point at an edge-connected ancestor); omit --graph for an empty draft. Slug defaults to the slugified name (globally unique). Node types: trigger | agent | integration | branch | gate (the walker runs trigger + agent + integration + branch + gate). Trigger node data = {schedule? | event?} (AT MOST ONE) — a manual trigger carries neither (fires only via `mc workflow run` / the canvas Run button). schedule = {cron? | intervalSec?, timezone?} makes an ACTIVE workflow fire on a cadence: the workflow-daemon enqueues a `cron`-triggered run whenever the schedule is due (reusing the profile scheduler's isDue). EXACTLY ONE of cron (a croner expression) or intervalSec (integer ≥ 60 — each fire is a paid run); timezone (IANA, e.g. America/New_York) applies to cron. The due-anchor is the workflow's last cron run, or — before its first fire — its updatedAt, so a freshly-activated cron waits for its next real instant instead of firing instantly; the single-flight guard suppresses a fire while a run is already queued/running. A workflow only cron-fires when status=active (`mc workflow activate`). event = {source?, types?} makes an ACTIVE workflow fire from an EXTERNAL webhook: POST /api/workflows/<slug>/webhook, authed by an HMAC over the RAW body — X-Hub-Signature-256: sha256=HMAC-SHA256(WORKFLOW_WEBHOOK_SECRET, body) (a single global env secret, GitHub-compatible, NEVER stored in the DB). A valid signature enqueues a `event`-trigger queued run carrying the JSON body in run.context (the workflow-daemon walks it; the web tier never spawns). The optional types allowlist is matched against the X-GitHub-Event (fallback X-Event-Type) header — empty/absent fires on any authenticated POST. Deliberate non-fires (inactive / not an event trigger / event-type filtered-out / single-flight in progress) return 200 {fired:false} so the sender doesn't retry; only a bad signature (401) / bad JSON (400) / unknown slug (404) are errors. `mc workflow webhook-url <slug>` prints the URL + signing details. Agent node data = {prompt (required), profileSlug?, projectSlug?, responseSchema?, onError?}. Integration node data = {toolkit (required; a Composio catalog slug — linear|slack|…), action (required; one of that toolkit's allowedTools), arguments?, onError?} — a DETERMINISTIC Composio action with NO LLM (no runs row, no spawn); runs against the project's `active` connection for that toolkit (mc composio connect), capturing the action's response `data` for downstream refs as {{node.output.*}}. Branch node data = {cases (required; ordered [{name, when:{left, op, right?}}]), onError?} — a DETERMINISTIC condition pick, NO LLM (no runs row, no spawn). The FIRST case whose condition is true wins (none → the implicit 'else'); execution ROUTES to the outgoing edges whose sourceHandle (fallback: label) equals the winning case name, and any node left with no active incoming edge is recorded `skipped` (a reconvergence point with ≥1 active edge still runs — an OR-join). op ∈ eq|ne|gt|gte|lt|lte|contains|truthy|falsy (gt/gte/lt/lte numeric; eq numeric when both operands look numeric, else string; contains = array-membership or substring; truthy/falsy ignore right). left/right may carry {{nodeId.field}} refs resolved TYPE-PRESERVING (so {{a.output.score}} compares as a number); an unresolvable condition ref hard-fails the branch. Gate node data = {message?, onError?} — a HUMAN approval gate, NO LLM (no runs row, no spawn). The walker PAUSES the run here: the gate's step sits 'running'/awaiting (non-terminal, so its successors never become decidable) and the run settles to status `paused` (a NEW non-terminal run status — reaper-EXCLUDED so it waits for a human indefinitely; single-flight-COUNTED so it still blocks a duplicate run). An operator resumes with `mc workflow approve <runId> <nodeId> [--reject --reason <t>]` (or the canvas Approve/Reject button): approve completes the gate + the walk continues; --reject FAILS the gate step, so its onError (halt default | continue) applies — exactly like any failed node. The decision is recorded on the gate step; resume is a sync re-walk (CLI default) or a paused→queued requeue the existing workflow-daemon claims (--async / the web button — the web tier never spawns). DATA PASSING: a prompt OR an integration node's arguments may embed {{nodeId.field}} refs resolved from an upstream node's captured output — {{id.result}} (agent free text), {{id.output}} / {{id.output.foo.bar}} (an agent's responseSchema structured_output OR an integration node's response data, dotted-path), {{id.status}} (terminal status). In an integration node's arguments a string that IS a single ref ("{{i.output.count}}") resolves to the RAW TYPED value (number/object/array preserved); an embedded ref ("issue #{{i.output.id}}") string-splices. responseSchema (JSON Schema) → claude --json-schema → captured structured_output (the {{id.output}} substrate). The TRIGGER node exposes its run.context (e.g. a webhook body) as {{trigger.output.*}} (same `output` substrate, via the trigger step's stored data), so an event-triggered graph reads the incoming payload (e.g. {{trigger.output.issue.title}}). onError = halt (default; a failed node stops the run) | continue (keep walking — the run still ends failed). A ref that can't resolve at run time (source failed / no such field) hard-fails the consuming node.
mc workflow update <slug> [--graph <json|@file>] [--name <n>] [--description <d>]   # the CLI twin of canvas authoring: replace a workflow's graph (and/or rename/re-describe) + bump its version. A provided --graph is validated through the SAME validateGraph SSOT as create/run (DAG, exactly one trigger, agent prompt, {{ref}}-ancestor); omit --graph to keep the current graph. The canvas Workflows tab has an EDIT mode that does the same write via POST {action:'save'}: drag node types from the palette onto the pane, drag between handles to connect (isValidConnection = the `canConnect` SSOT, so a cycle / self-loop / edge-into-a-trigger can't be drawn), edit the selected node in the inspector (typed common fields + a `data (JSON)` escape hatch for responseSchema/arguments/cases/schedule), Backspace to delete, Save → validateGraph (an invalid graph surfaces the error inline, never persists). Editing is disabled while a run is queued/running/paused.
mc workflow run <slug> [--timeout <sec>] [--max-parallel <n>] [--allow-concurrent] [--async]   # run a workflow. DEFAULT = SYNCHRONOUS: this process owns the walk and blocks to completion (born 'running'; keep prompts short). --async = DURABLE: write a 'queued' run (lib-tier, NO spawn) and return immediately; the workflow-daemon (`npm run workflow-daemon`) claims it race-safe (queued→running) and executes it off-process — also exactly what the canvas Run button does. CONCURRENT WALK (ready-set scheduler): a node is decidable once ALL its predecessors are terminal (completed|failed|skipped — a WAIT-ALL join), so independent nodes (a fan-out's branches) run in PARALLEL while a merge node waits for every incoming branch; `--max-parallel <n>` caps in-flight nodes (default 4, sync path). Fan-out/merge are graph shapes (a node with many out-/in-edges), NOT node types. A trigger completes immediately; an agent node opens a real run (mc run start) and spawns the resolved profile via the shared daemon runner; integration/branch nodes are deterministic (no run). RUN-ONLY visibility: cost/heartbeat/fleet-feed/cancel come from that run row — NO claimable task is created (so the auto-claim daemon can't race it). A failed node halts the workflow (stop launching new nodes; in-flight ones drain) unless its onError='continue'. Graph snapshotted onto the run; single-flight guard refuses a second run while one is queued OR running unless --allow-concurrent. A dead walker's 'running' run is reconciled to 'failed' by the reaper (heartbeat); a 'queued' run survives a daemon restart and is claimed later.
mc workflow status <runId>                   # a workflow run + its per-node step rows (each agent step links its runs id + captured output); run status = queued|running|completed|failed|cancelled
mc workflow cancel <runId>                   # cancel a workflow run. A 'queued' (never started) OR 'paused' (gate-awaiting, not actively walking) run is marked 'cancelled' outright; a 'running' run gets cancelRequested, propagated to the in-flight agent node's run (kill-switch)
mc workflow approve <runId> <nodeId> [--reject] [--reason <t>] [--async]  # decide a PAUSED gate (slice 9a) and resume the run. Default = synchronous (record the decision + flip paused→running + walk to the next terminal/paused state); --async records the decision + requeues paused→queued for the workflow-daemon (the canvas Approve/Reject button uses this path — the web tier never spawns). --reject fails the gate step (its onError then halts/continues). Refuses a run that isn't paused (CONFLICT)
mc workflow activate <slug>                  # set a workflow's status to active — a cron/interval trigger then fires on its schedule via the workflow-daemon
mc workflow pause <slug>                     # set a workflow's status to paused
mc workflow webhook-url <slug>               # print the external webhook URL (/api/workflows/<slug>/webhook) + HMAC signing details (X-Hub-Signature-256 over the raw body, secret = WORKFLOW_WEBHOOK_SECRET env) for an event-triggered workflow; readonly
mc profile list [--enabled] [--runtime claude-code|exec] [--schedulable]   # --schedulable = enabled + scheduled check-ins on (the scheduler's scan)
mc profile get <slug>
mc profile add --slug <s> --name <n> [--runtime claude-code|exec] [--model <m>] [--fallback-model <m>] [--daily-budget-micros <n>] [--provider <p>] [--base-url <u>] [--permission-mode plan|acceptEdits|bypassPermissions|default] [--skills a,b] [--mcp-config <json|@file>] [--allowed-tools <csv>] [--disallowed-tools <csv>] [--append-system-prompt <t>] [--env K=V ...] [--exec-template <cmd>] [--match-project <csv>] [--match-category <csv>] [--match-kind <csv>] [--match-label <regex>] [--priority <n>] [--default] [--disabled] [--schedule-enabled] [--schedule-disabled] [--schedule-project <slug>] [--schedule-interval <sec>] [--schedule-cron <expr>] [--schedule-timezone <tz>] [--check-in-prompt <t|@file>]  # capability bundle + auto-routing rules. runtime=exec needs --exec-template (drives non-Claude models). Secrets are ${ENV} placeholders only — never stored. fallbackModel → claude --fallback-model (resilience) + the budget-downgrade target; once dailyBudgetMicros of THIS profile's same-UTC-day run cost is exceeded the daemon renders fallbackModel. SCHEDULED CHECK-INS (≠ the liveness heartbeat): an enabled schedule needs --schedule-project + exactly one of --schedule-interval/--schedule-cron; the scheduler daemon wakes the profile on that cadence, spawns a run in the project's repo with --check-in-prompt, and the agent self-serves that project's queued tasks. --schedule-interval has a floor (>=60s; each check-in is a paid run). --schedule-cron is evaluated in --schedule-timezone (IANA zone, e.g. America/New_York); omit it and the cron resolves in the daemon's local time (often UTC under launchd -> wrong hour). Pass "" to --schedule-interval/--schedule-cron to clear a trigger.
mc profile update <slug> [any add flag; only provided change] [--enabled]
mc profile set-default <slug>                # the single global fallback when no rule matches (idempotent)
mc profile checked-in <slug> [--status ok|fail]   # the scheduler records a check-in: advances last_check_in_at; ok resets / fail increments consecutive_failures (auto-pauses the schedule after 3)
mc profile rm <slug> --yes
mc profile resolve [--project <slug>] [--task <id>] [--label <text>] [--kind custom|integration]  # preview which profile auto-routing picks (matchRules → priority → default)
mc run start --agent <label> [--project <slug>] [--profile <slug>] [--title <t>] [--source hook|cli|cron|manual] [--model <m>] [--session-id <id>] [--work-dir <dir>] [--id <uuid>]
mc run end <id> <status> [--tokens-in <n>] [--tokens-out <n>] [--cache-read <n>] [--cache-write <n>] [--cost-micros <n>] [--agent <label>]
mc run list [--active] [--agent <label>] [--limit <n>]
mc run get <id>                              # one run + its event trail
mc run cancel <id>                           # request cancellation (sets cancel_requested; enforced by the PreToolUse kill-switch hook when installed)
mc event add <summary...> --type <t> [--level debug|info|warn|error] [--project <slug>] [--task <id>] [--run <id>] [--agent <label>]
mc event list [--project <slug>] [--run <id>] [--level <min>] [--limit <n>]
mc spend [--group-by project|agent|day|run] [--since <iso>] [--until <iso>] [--project <slug>] [--agent <label>] [--profile <slug>] [--limit <n>]  # cost rollup over runs (sums runs.cost_micros)
```
