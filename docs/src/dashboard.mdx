# What you see: the dashboard

When you sign in, the website has five top-level tabs. Each project page also gains extra tabs as you connect external services.

<Frame caption="Tab 1 · Overview — headline numbers, the live Fleet Glance strip, and recently active projects.">
  ![Mission Control Overview tab](/screenshots/overview.png)
</Frame>

<CardGroup cols={2}>
  <Card title="Tab 1 · Overview" icon="gauge">
    The "at a glance" page. A row of headline numbers — how many projects, how many launched,
    how many clients — plus a **Fleet Glance** strip showing the 6 most recently active AI runs
    (with a link to the full Mission view), and a list of recently touched projects.
  </Card>
  <Card title="Tab 2 · Projects" icon="folder-kanban">
    The main work surface: every project in one sortable, searchable register — name, type,
    tech used, setup status, number of tasks, last activity. You can add projects and tasks
    right here.
  </Card>
  <Card title="Tab 3 · Board" icon="square-kanban">
    A drag-and-drop Kanban across every active project — one swimlane per project, with To Do /
    In Progress / Done columns. Drag a card to move or re-prioritise its work, and the order you
    set steers what the AI picks up next. (Each project page has the same board too — as a
    **List ⇄ Board** toggle inside its Tasks tab.)
  </Card>
  <Card title="Tab 4 · Mission" icon="radar">
    The live ops room. A "fleet" strip shows AI agents working right now, with a pulsing dot for
    the live ones, and an activity feed streams events as they happen — like a flight tracker for
    your AI.
  </Card>
  <Card title="Tab 5 · Spend" icon="dollar-sign">
    The money view. Total spent, number of runs, tokens used, and a breakdown you can group by
    project, by agent, by day, or by individual run.
  </Card>
</CardGroup>

<Info>
**Where the page gets its numbers.** When you load a tab, the website reads straight from the
database and builds the page on the server before sending it to you — so it's fast and always
current. When you make a change, it's saved to the database and the page refreshes its data.
The Mission tab goes further: it quietly re-checks for new activity every few seconds, so it
stays live.
</Info>

## The board: drag work, steer the agents

The **Board** is both a window into what the agents are doing and the lever you use to decide
what they do next. It comes in two shapes that share the same three columns — **To Do**,
**In Progress**, and **Done**.

<CardGroup cols={2}>
  <Card title="The overall board" icon="square-kanban">
    The top-level **Board** tab. One **swimlane** per active project, stacked with the live fleet
    strip on top — so you see every project's queue at once. You can re-order cards *within* a
    project, but a card can't jump to a different project (its lane is fixed).
  </Card>
  <Card title="The per-project board" icon="folder-kanban">
    Each project's **Tasks** tab carries a **List ⇄ Board** toggle: the same tasks as a flat list
    or as that one project's Kanban columns, with a selector for how far back the **Done** column
    reaches. (There's no longer a separate per-project Board tab — it's a view of the Tasks tab.)
  </Card>
</CardGroup>

<Frame caption="Tab 3 · Board — one swimlane per project across To Do / In Progress / Done, with the live fleet strip on top.">
  ![Board tab](/screenshots/board.png)
</Frame>

Two kinds of drag do two different things:

- **Drag a card between columns** to change its status — `To Do → In Progress → Done` (and back).
- **Drag a card up or down within a column** to re-order it — that's the priority order.

<Note>
**Re-ordering the To Do column steers the autopilot.** The position you give a card *is* its
priority: the auto-claim daemon takes the **top** of To Do next. Drag a card to the top and it
becomes the next thing an AI worker picks up — see [Autonomy](/autonomy). A pure re-order is
silent — it doesn't count as activity on the project, it just changes who's next in line.
</Note>

<Warning>
**A card an agent is actively working is locked.** While a task is claimed by a live run, its
card can't be dragged — you can't yank work out from under a running agent. It frees up the
moment that run ends. Moves are also optimistic: the card jumps immediately, and if the write
loses a race (someone else moved it, or it was just claimed) the board quietly snaps it back on
its next refresh.
</Warning>

## Per-project tabs

Every project page always shows the same core tabs (Overview, Tasks, MCP, Workflows, Activity) — the
Tasks tab carries the List ⇄ Board toggle, so the board is a view there rather than its own tab.
Three more — Errors, Email, and Revenue — appear automatically once you set the matching project
field, and each pulls live data from an external service every time you open it. The table below
covers those three plus the always-present MCP and Workflows tabs (which read from the database, not
a live service).

<Frame caption="Fig. 07 — Per-project tabs and where their data comes from.">
  ![Per-project tabs diagram](/figures/fig-07-tabs.svg)
</Frame>

| Tab | Appears when… | What you see | Data source |
|-----|---------------|--------------|-------------|
| **Errors** | `sentryProjectSlug` is set | Up to 25 unresolved Sentry issues from the last 24 h — level, title, culprit, event and user counts, last-seen time, each linking to Sentry; plus a summary of total unresolved and 24 h event volume. If the token is missing the tab shows a setup hint instead of erroring. | Live Sentry REST API |
| **Email** | `domain` is set | MX, SPF, and DMARC DNS checks — pass/fail with the raw record value. Auto-detects the email provider (Zoho, Google Workspace, Microsoft 365, Proton, Fastmail, iCloud) and also shows any manually set `emailProvider` / `emailAddress`. | Live DNS lookup (no token needed) |
| **Revenue** | `stripeSite` is set | MRR per currency, active subscription count (shown as N+ when truncated at 1,000), and a table of active subscriptions (customer, amount, billing interval, created date). One Stripe account is shared across all projects; subscriptions are filtered by `metadata.site`. | Live Stripe Search API |
| **MCP** | Always present | This project's MCP servers (both sources), a searchable browse of Composio's full live catalog, and a form to add a remote MCP server. Each connected server shows its status — Active, Initializing, Error, Expired, or Disconnected — with the matching action. After you authorize a new Composio connection, any still-initializing toolkit is polled automatically. See [MCP servers](/mcp) for the full story. | Database read (no live Composio call on GET) |
| **Workflows** | Always present | This project's [workflows](/workflows) on a React Flow canvas — pick one from the rail and its graph renders with a live per-node status overlay during a run. **+ New** creates one and drops you onto a blank canvas; **Edit** mode authors the graph (drag nodes from a palette, connect them, edit each in an inspector, Save); **Run** enqueues an execution; a paused run shows an **Awaiting approval** panel with Approve / Reject. | Database read + the workflow daemon executes |

<Note>
**Setting the project fields.** Use `mc project update` to enable each tab:

```bash
mc project update my-app --sentry-project my-app-slug   # unlocks Errors tab
mc project update my-app --domain my-app.com             # unlocks Email tab
mc project update my-app --stripe-site site_abc123       # unlocks Revenue tab
```
</Note>

<Info>
**Environment variables required for live tabs.** The Errors tab needs `SENTRY_AUTH_TOKEN` and
`SENTRY_ORG` (and optionally `SENTRY_BASE_URL` for self-hosted Sentry). The Revenue tab needs
`STRIPE_SECRET_KEY`. The Email tab uses the server's DNS resolver — no token required.
</Info>
