# Giving each agent the right tools

Named, reusable bundles of capabilities and routing rules that decide *how* an agent is spawned
for a task.

By default every auto-claimed worker is identical: the same model, the same plan-only safety
mode, the same blank toolkit. **Agent profiles** change that. A profile is a saved recipe — a
model, a permission posture, a set of skills, MCP servers, a tool policy, a persona, even a
non-Claude runtime — plus the **rules that decide when to use it**. When the daemon picks up a
task, it finds the matching profile and spawns the worker to that recipe.

Think of it like staffing: a "release engineer" profile (cautious, can run deploys, points at
your CI tools) versus a "docs writer" profile (a cheaper model, read-mostly, no deploy access).
The right one shows up for the right job, automatically.

</Frame>

## What's in a profile

| Field | What it controls |
|---|---|
| **runtime** | How the worker is launched: `claude-code` (rich `claude -p` flags) or `exec` (any command — the door to non-Claude models). |
| **model** · **permission mode** | Which model answers, and how much it's allowed to do without asking (`plan`, `acceptEdits`, `bypassPermissions`, `default`). |
| **skills** | Skills the agent must have — an enforced contract, not just steering. Resolved on disk at spawn (user, work-dir, and plugin skills); a missing one fails the run loudly. See [Skills](#skills). |
| **MCP servers** | Model Context Protocol tools the agent can call — by `stdio` command or `http`/`sse`/`ws` URL. The real "powers." |
| **allowed / disallowed tools** | A tool allow-list and deny-list. |
| **append system prompt** | A persona or extra standing instructions. |
| **env** | Environment variables for the worker — secrets as `${ENV}` placeholders only (see below). |
| **fallback model** · **daily budget** | Cost-aware routing — resilience plus a spend cap (see [Cost](/cost)). |
| **match rules** · **priority** · **default** | When this profile applies, how ties break, and the global fallback. |
| **scheduled check-in** | An optional cadence (interval or cron) that wakes this profile on its own to work a bound project's queue (see [below](#scheduled-check-ins)). |

## MCP auto-feed

When the daemon spawns an agent it also fetches the project's active [MCP servers](/mcp) — both
Composio-brokered toolkits and direct remote servers — and merges them into the agent's MCP server
set automatically, with no extra profile configuration required.

How it works:

- Each active server becomes an MCP server in the agent's config — a Composio connection keyed
  `composio-<toolkit>` (e.g. `composio-linear`, `composio-slack`), a remote server keyed by its name.
- These servers are merged **under** the profile's own `mcpServers`. If the profile already
  declares a key that matches, the **profile wins** — you can always override an auto-fed server
  with a hand-crafted one.
- The API key is carried as an `x-api-key` header using the `${COMPOSIO_API_KEY}` placeholder,
  resolved from the daemon's host environment at spawn. **It is never stored in the database.**
- If the project has no active connections, or the lookup fails, the run still spawns — just
  without the auto-fed servers (non-fatal).
- When no profile matches the task (the null-profile path), the daemon still feeds the active
  connections and passes `--strict-mcp-config` so the profileless agent sees exactly those
  servers and nothing else.

<Note>
**Secrets are resolved at spawn, never stored.** The `${COMPOSIO_API_KEY}` placeholder (and any
other `${ENV}` placeholders in your profile's MCP config) is resolved from the daemon's
environment at the moment the agent is launched. The resolved value is written to a `0600` temp
file and is never visible in the database, in `ps` output, or on disk in the clear.
</Note>

See [MCP servers](/mcp) for how to connect a project to a toolkit or a remote server and manage its
active connections.

## Skills

<Note>
This section covers how a *profile* declares and enforces skills. For the skill subsystem as a whole —
where skills live, the resolvable catalog, and discovering/installing new ones from the
[skills.sh](https://skills.sh) registry — see the dedicated [Skills](/skills) page.
</Note>

A profile's `skills` array is an **enforced contract**, not a hint. At spawn the daemon resolves
every declared skill on disk; if any one doesn't resolve, the run **fails loudly** (a
`skill.unresolved` event names the skill and why) rather than launching an agent whose declared
capability is silently absent. The resolved skills still get a steering line in the system prompt —
discovery and steering are complementary.

A profile can declare three kinds of skill:

| Source | Declared as | Resolves from |
|---|---|---|
| **user** | `ship` | `~/.claude/skills/<name>/SKILL.md` |
| **work-dir** | `deploy-helper` | the project repo's `<repo>/.claude/skills/<name>/SKILL.md` |
| **plugin** | `compound-engineering:ce-work` | an enabled, installed Claude Code plugin |

The colon is the switch: a bare name (`ship`) resolves against the user and work-dir directories; a
`plugin:skill` name resolves against plugins only. To make both filesystem locations discoverable,
a skill-declaring profile spawns with `--setting-sources user,project`.

**Plugin skills** resolve only when the plugin is *both* enabled (in the union of your user and the
project's `.claude/settings.json` `enabledPlugins`) *and* installed — so a declared plugin skill can
fail for one of three precise reasons, each pointing at its fix:

| Reason | Meaning | Fix |
|---|---|---|
| `plugin-disabled` | The plugin is installed but not enabled. | Enable it in `enabledPlugins`. |
| `plugin-not-installed` | No install for that plugin. | Install the plugin. |
| `skill-not-found` | The plugin is enabled and installed but has no such skill. | Check the skill name. |

Plugin skills need **no extra spawn configuration** — enabled plugins are already visible to a
headless agent through your user settings.

**Preflight before you schedule.** Two read-only commands resolve against the same locations the
spawn does, so you can catch a misconfigured profile before an (often unattended) run hard-fails:

```bash
mc skill list                      # every resolvable skill — user, work-dir, and plugin (source-tagged)
mc skill list --project myapp      # also include that project's work-dir + project-enabled plugins
mc profile resolve --project myapp # the matched profile's declared skills: resolved (with source) or the reason
```

Because the catalog and the spawn-time check share one resolver, anything `mc skill list` shows is
guaranteed to resolve at spawn — and anything it omits will not.

**Need a skill you don't have yet?** Discover and install one from the
[skills.sh](https://skills.sh) registry, then declare it on a profile:

```bash
mc skill search react              # search the registry (each hit flagged installed or not)
mc skill add owner/repo@some-skill # fetch it from GitHub into ~/.claude/skills/<slug>/
mc skill list                      # it now resolves like any hand-placed skill
```

`mc skill search` reads the public skills.sh registry (no credentials); `mc skill add` fetches the
skill's files from its public GitHub repo and writes them into `~/.claude/skills/` — the one place
`mc` mutates your Claude config, guarded by name, path-traversal, and frontmatter checks. Once
installed, the skill flows through the same derived catalog and enforced-at-spawn contract as every
other skill. See the [CLI reference](/reference/cli#skills) for the full command surface.

## How the right profile gets picked

Routing is rule-based and runs every time the daemon claims a task.

<Steps>
  <Step title="Match on the task's context">
    Each profile can declare **match rules** — any mix of project slugs, project categories, and a
    label regex. A profile applies only when *every* rule it sets is satisfied (they're ANDed). A
    profile with no rules never matches by rule — it can only be the default.
  </Step>
  <Step title="Break ties by priority">
    If several profiles match, the one with the highest **priority** wins; the oldest is the
    tiebreaker, so the choice is stable.
  </Step>
  <Step title="Fall back to the default">
    If nothing matches, the single **default** profile is used. If there's no default either, the
    daemon spawns its plain built-in worker — exactly as before profiles existed.
  </Step>
</Steps>

You can preview the decision without running anything:

```bash
mc profile resolve --project myapp --label "fix: crash on save"
```

## Two runtimes: Claude and everything else

Claude Code is Claude-only — pointing it at a different base URL changes *where* requests go, not
*which* model answers. So a profile carries a first-class `runtime` field:

- **`claude-code`** renders rich `claude -p` flags: `--model`, `--permission-mode`,
  `--append-system-prompt` (persona + a steering line for the resolved skills),
  `--allowedTools` / `--disallowed-tools`, `--mcp-config` (with `--strict-mcp-config` so the
  profile's server set is authoritative), and — when the profile declares skills —
  `--setting-sources user,project` so the agent discovers both skill locations (see [Skills](#skills)).
- **`exec`** renders a command template you supply, with `${MODEL}`, `${PROMPT}`,
  `${MCP_CONFIG}`, and `${FALLBACK_MODEL}` tokens filled in — so a profile can drive a non-Claude
  model through any runner (an OpenAI-compatible gateway, a local model, whatever you script).

<Warning>
**Secrets are placeholders, never stored.** Anywhere a secret would go — an `env` value, an MCP
header like `Authorization` — you store a `${ENV_VAR}` placeholder, not the secret itself. The
daemon resolves it from its own environment at spawn and writes any rendered MCP config to a
`0600` temp file, so a resolved key never lands in the database, in `ps` output, or on disk in the
clear. If a referenced variable isn't set, the run fails fast with a clear message rather than
launching half-credentialed.
</Warning>

## Cost-aware routing

Two profile fields keep a model from running away with your budget:

- **fallback model** maps to claude's `--fallback-model` — if the primary is overloaded or
  unavailable, it transparently retries on the cheaper one.
- **daily budget** (micro-dollars) is a spend cap on *this profile's* same-day run cost. Once the
  day's spend crosses the cap, the daemon **downgrades** new runs to the fallback model and logs
  an event saying why. It reads real spend (`mc spend --profile`), not a guess at task difficulty.

Both are optional; a profile with neither always runs its primary model. See [Cost](/cost).

## Scheduled check-ins

A profile usually waits to be *matched* to a task. Flip on a **scheduled check-in** and it stops
waiting: the [scheduler daemon](/reference/operating#the-scheduler-daemon) wakes the profile on a
cadence you set, opens a run in a project's repo, and lets it work that project's queue. It's how
you stand up a recurring agent — "every weekday at 9am, triage the open issues in `myapp`." The
mechanics of a wake-up live in [Autonomy](/autonomy#two-ways-work-starts-claimed-and-scheduled);
here's how you arm one.

A schedule has four parts:

| Setting | Flag | What it means |
|---|---|---|
| **Project** | `--schedule-project <slug>` | The repo the check-in runs in, and the queue it serves. Required to enable. |
| **Cadence** | `--schedule-interval <sec>` **or** `--schedule-cron <expr>` | Fire every N seconds, **or** on a cron expression — exactly one. Interval has a **60-second floor**; each check-in is a paid run. |
| **Timezone** | `--schedule-timezone <tz>` | The IANA zone (e.g. `America/New_York`) the cron is read in. Omit it and cron resolves in the daemon's local time — usually UTC under launchd, so set it. |
| **Mission** | `--check-in-prompt <text\|@file>` | The standing instruction handed to the agent on every wake-up. |

```bash
# Triage myapp's issue queue every weekday at 9am Eastern
mc profile add --slug triager --name "Issue triager" \
  --model claude-sonnet-4-6 --permission-mode acceptEdits \
  --schedule-enabled --schedule-project myapp \
  --schedule-cron '0 9 * * 1-5' --schedule-timezone America/New_York \
  --check-in-prompt @prompts/triage.md
```

Turn it on or off with `--schedule-enabled` / `--schedule-disabled`; clear a trigger by passing
`""` to its flag. A self-serving check-in needs the `mc` tool — out of the box the scheduler grants
`Bash(mc:*)` and runs in `acceptEdits`, so you don't have to widen `--allowed-tools` yourself unless
you want to.

<Note>
**Auto-pause keeps a broken mission cheap.** The scheduler records the result of every check-in
(`mc profile checked-in`). Three consecutive failures and the schedule **auto-pauses** itself — the
Profiles row shows a red `PAUSED` badge with the failure count. Edit the profile and re-enable to
clear it.
</Note>

## Managing profiles

Profiles are global (not per-project) and slug-addressed, like projects. Drive them from the CLI
or the **Profiles** tab in the dashboard — both write through the same core, so they can't drift.

```bash
# A claude-code profile that handles deploy tasks with a cautious posture + a cost cap
mc profile add --slug releaser --name "Release engineer" \
  --model opus --permission-mode acceptEdits \
  --allowed-tools "Bash,Edit" --skills deploy,canary,compound-engineering:ce-work \
  --match-label '^(deploy|release):' --priority 5 \
  --fallback-model claude-sonnet-4-6 --daily-budget-micros 5000000

mc profile set-default <slug>     # the global fallback when no rule matches
mc profile list                   # all profiles
mc profile resolve --project myapp  # preview which one auto-routing picks
```

Every run records which profile (and which model) it actually used, so the [Spend](/cost) and
activity views can tell you exactly what each profile is costing and doing.
