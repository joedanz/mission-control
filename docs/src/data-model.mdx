# How the information is organized

Four "drawers" in the filing cabinet, neatly linked together.

To make sense of the system you only need to understand four things it keeps track of. Each
one "belongs to" the one above it, like folders inside folders. A fifth â€” the **agent profile**
â€” sits off to the side: reusable config a run can point at, rather than a folder in the stack.

| Drawer | In plain words | What it remembers |
|---|---|---|
| **Project** | A thing you're building â€” a website, an app, a client job. | Name, category (`internal`, `open_source`, `client`), status, tech, links, priority. |
| **Task** | One item of work inside a project â€” a custom to-do, or a standard "integration" like payments. | Label, status (`todo` â†’ `in_progress` â†’ `done`), its position on the board (which sets its priority in the queue), and â€” if an AI grabbed it â€” which run claimed it and when the claim expires. |
| **Run** | One work-session by one AI agent. Starts when it begins, ends when it finishes. | Which agent, its status (`running` / `completed` / `failed` / `abandoned`), tokens used, dollar cost, and a "heartbeat" proving it's alive. |
| **Event** | A single timestamped line in the activity log â€” the diary of everything that happened. | What happened, how important it is, and links to the related run, task, and project. |
| **Agent profile** | A reusable recipe for *how* an agent runs â€” its model, skills, tools, and routing rules. A run records which one it used. | Model, runtime, permission mode, skills, MCP servers, match rules, fallback model + budget. See [Agent profiles](/profiles). |
| **Composio connection** | A project's live link to one third-party toolkit (Linear, Slack, and so on) brokered by Composio. | **Workflow** | A defined *flow* of work â€” a graph of nodes (a trigger, AI agents, integration calls, branches, approval gates) you draw on a canvas. It composes the pieces above rather than replacing them. | The node graph and its version, plus â€” per execution â€” a **workflow run** that snapshots the graph and a **step** per node (an agent node's step links the [run](/data-model) it spawned). See [Workflows](/workflows). |

<Note>
**No two agents can grab the same task.** When an AI claims a task, it stamps the task with its
own ID *and* an expiry time in a single, indivisible database operation. If two agents reach
for the same task at the same instant, the database lets only one win â€” the other is politely
told "already taken." That's what makes the autopilot safe to run with many workers at once.
Engineers call this *optimistic concurrency*; the `version` and `claim_expires_at` fields make
it work.
</Note>
