# The star feature: AI that works on its own

A closed loop where AI finds, does, and reports its own work — no human in the middle.

Most dashboards just *show* you information. Mission Control can also **do the work**. A
background dispatcher — the **auto-claim daemon** — watches the task queue. The moment a task
is free, it launches an AI worker to handle it, no human needed. Here is the full cycle.

<Frame caption="Fig. 03 — Autonomous loop. 01–03 assign work, 04–06 do it and tidy up — then it repeats.">
  ![Autonomous task loop](/figures/fig-03-run-loop.svg)
</Frame>

<Steps>
  <Step title="Find a free task">
    The dispatcher asks the database for the next available task. It only considers tasks that are
    unclaimed, or whose previous claim has expired — and it takes them in **board order**: the top
    of a project's To Do column first, oldest-first only to break ties. Drag a card up the
    [board](/dashboard#the-board-drag-work-steer-the-agents) and you've changed what it claims next.

    ```bash
    mc task next --project myapp
    ```
  </Step>
  <Step title="Claim it — safely">
    It stamps the task as "taken" in one atomic step. If another worker got there a millisecond
    earlier, this one is told "already claimed" and simply moves on. No double-work, ever.

    ```bash
    mc task claim <task-id> --run <run-id>
    ```
  </Step>
  <Step title="Spawn an AI worker">
    The dispatcher launches a fresh AI assistant pointed at that task, in "plan" mode by default
    for safety. The task description is treated as untrusted text, so a sneaky task can't hijack
    the agent.

    If an [agent profile](/profiles) matches the task, the worker is spawned to that recipe instead
    — its model, skills, MCP tools, permission posture, or even a non-Claude runtime. Otherwise the
    plain default worker runs, exactly as described here.
  </Step>
  <Step title="The agent works — and beats a heartbeat">
    As the AI uses tools — reading files, writing code — it automatically pings the database every
    step. That heartbeat is how we know it's still alive and making progress.
  </Step>
  <Step title="Report results">
    When the AI finishes, it tallies how many tokens it used and what that cost, then marks its
    run `completed` (or `failed`).

    ```bash
    mc run end <run-id> completed
    ```
  </Step>
  <Step title="Settle the task">
    If the run succeeded, the task becomes `done`. If it failed or was stopped, the claim is
    released and the task drops back into the queue for another attempt. Then the loop starts over.
  </Step>
</Steps>

<Info>
**There's no human in any of those six steps.** Tasks flow in — you add them, or they're
imported from GitHub issues — the system assigns and does them, and the results flow back to
your dashboard. Your job shifts from *doing* the work to *steering and reviewing* it.
</Info>

## Two ways work starts: claimed and scheduled

The loop above is **reactive** — it fires the instant a task lands in a project's queue. There's a
second trigger that's **proactive**: an [agent profile](/profiles#scheduled-check-ins) can wake
*itself* on a schedule, drop into a project's repo, and work that project's queue without anyone
adding a task first.

A second always-on daemon — the **scheduler** — watches every profile that has scheduled check-ins
turned on. When a profile comes due, the scheduler runs a tight version of the same loop:

<Steps>
  <Step title="Start a run in the bound project's repo">
    It opens a fresh run (`source: cron`) in the repo of the profile's **schedule project**, handing
    the agent the profile's standing **check-in prompt** as its mission. The check-in clock advances
    at spawn, so a slow run never double-fires; a crash before spawn just retries next tick.
  </Step>
  <Step title="Pre-claim one task for the run">
    Before the agent even starts, the scheduler peeks the project's queue (`mc task next`) and
    claims the top task to this run. So task tracking no longer depends on the agent remembering to
    call `mc` — a clean run auto-completes the claimed task, a failed one releases it back.
  </Step>
  <Step title="Hand the agent the keys to self-serve">
    Every check-in is granted the `mc` tool (`Bash(mc:*)`) and runs in `acceptEdits` out of the box
    — a check-in has to *act*, not just plan. If the profile allows more than one task per check-in,
    the prompt invites the agent to drain the rest of the queue in the same session.
  </Step>
</Steps>

If a check-in fails, the profile's **consecutive-failure** counter ticks up; after three in a row
the scheduler **auto-pauses** the schedule (and flags it on the Profiles row) so a broken mission
can't burn money on a loop. Edit the profile and re-enable to clear it.

See [Agent profiles](/profiles#scheduled-check-ins) to set a cadence, and
[Operating](/reference/operating#the-scheduler-daemon) for the daemon itself.
