# The database, column by column

Eight working tables and the auth tables, defined with Drizzle ORM on Neon Postgres.

Every `id` is a UUID stored as text. Enumerated values are split deliberately into two kinds:
**hard `pgEnum` types** for sets that almost never change, and **plain text validated in the
app/CLI layer** for "churny" sets that do â€” so adding a status never needs a database migration.

## Indexes that matter

| Index | Purpose |
|---|---|
| `tasks_claimable_idx` | Partial over only `status='todo' AND kind='custom'` rows, ordered `(sort_order, created_at)` â€” priority by board position, then FIFO. The daemon polls this on a loop, so a board re-order changes what it picks up next. |
| `runs_status_heartbeat_idx` | The reaper's index â€” finds `running` runs whose heartbeat went stale. |
| `events_order_idx` | `(created_at, seq)` â€” the activity feed's replay order. |
| `events_idempotency_uq` | Partial unique on `idempotency_key` â€” the dedupe guarantee. |
| `tasks_project_integration_uq` | One integration row per type per project (keeps the grids from double-counting). |
| `agent_profiles_slug_uq` | Unique profile slug â€” the handle you address with `mc profile`. |
| `agent_profiles_default_uq` | Partial unique over `is_default = true` â€” enforces *at most one* global default profile. |
| `agent_profiles_schedulable_idx` | Partial over rows with `schedule_enabled` â€” the scheduler's scan for profiles that might be due. |
| `composio_connections_project_toolkit_uq` | UNIQUE on `(project_id, toolkit_slug)` â€” enforces the intentional one-connection-per-(project, toolkit) invariant. Two connections for the same toolkit in the same project are an explicit non-goal; dropping this index requires a per-connection `user_id` redesign. |

<Note>
**Why two agents never do the same task.** A claim is a **single conditional write**: "set
`claimed_by_run_id` and `claim_expires_at` on this task *only if* it's unclaimed or its claim
has expired." Because it's one indivisible statement, concurrent agents can't both win â€” exactly
one row update succeeds; the rest get zero rows back and surface a `CONFLICT`. Claiming is
**orthogonal to status**: it never moves the task to `in_progress`. This is textbook *optimistic
concurrency*, and it's what makes the autopilot safe to run many-at-once.
</Note>
