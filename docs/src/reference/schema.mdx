# The database, column by column

Eight working tables and the auth tables, defined with Drizzle ORM on Neon Postgres.

Every `id` is a UUID stored as text. Enumerated values are split deliberately into two kinds:
**hard `pgEnum` types** for sets that almost never change, and **plain text validated in the
app/CLI layer** for "churny" sets that do вЂ” so adding a status never needs a database migration.

## projects вЂ” "a thing you're building"

| Column | Notes |
|---|---|
| `id` В· name В· `slug` | UUID, display name, and a unique URL-safe handle (the address you pass to `mc`). |
| category В· status В· accent | `category`/`accent` are pgEnums; `status` is validated text. |
| tech_stack | `jsonb` string array, indexed with a GIN index for tech-based filtering. |
| repo_path В· repo_url В· live_url | Local checkout (daemon target), GitHub URL (issue import), and the deployed site. |
| priority В· target_date В· notes | Planning metadata. |
| archived В· sort_order | Soft-hide flag and manual ordering. |
| sentry_project_slug | Nullable text. Sentry project slug вЂ” when set, enables the Errors tab. |
| email_provider В· email_address | Nullable text. Manual provider label and primary email address вЂ” when set, enables the Email tab. |
| stripe_site | Nullable text. Stripe `metadata.site` value вЂ” when set, enables the Revenue tab. |
| created_at В· updated_at В· last_activity_at | Timestamps; `last_activity_at` drives the "recently touched" list. |

## tasks вЂ” "one item of work"

| Column | Notes |
|---|---|
| `project_id` | FK to `projects`, `ON DELETE CASCADE` вЂ” delete a project, its tasks go too. |
| label В· notes В· kind | `kind` is `custom` or `integration`. |
| status В· integration_status | Custom workflow (`todoв†’in_progressв†’done`) vs the tri-state for integration tasks (`needed/pending/done`). |
| version | Optimistic-concurrency token, bumped on every status/claim write вЂ” the collision signal. |
| sort_order | Manual board position within a column; lower comes first. Drives the queue head (see `tasks_claimable_idx`) so a board re-order steers what the daemon claims next. |
| claimed_by_run_id | FK to `runs`, `ON DELETE SET NULL` вЂ” a deleted/abandoned run frees the claim. |
| claimed_at В· claim_expires_at | When the claim was taken and when it auto-releases (the backstop). |
| completed_at | Set when the task reaches `done`. |

## runs вЂ” "one AI work-session"

| Column | Notes |
|---|---|
| `id` | Client-supplied by the hook, so a retried `run.start` upserts the same row. |
| agent_label В· parent_run_id | Free-text agent name; self-FK builds sub-agent trees. |
| project_id В· title В· source В· model | Context; `source` is `hook\|cli\|cron\|manual` (`cron` = a scheduled check-in). `model` is the model that actually ran (the daemon writes the budget-chosen one). |
| agent_profile_id | FK to `agent_profiles`, `ON DELETE SET NULL` вЂ” which profile this run used; deleting the profile leaves the run intact. |
| status | `running в†’ completed \| failed \| abandoned`. "Live/idle" is *derived* from the heartbeat, never stored. |
| tokens_in/out В· cache_read/write_tokens В· cost_micros | `bigint` cumulative totals (cache split out вЂ” it dominates Claude cost); cost in micro-dollars. |
| session_id В· work_dir В· transcript_ref | Claude Code's session id, the hook's cwd, and a *path* to the transcript вЂ” never its body. |
| cancel_requested | The kill-switch flag; surfaced on the heartbeat response and enforced by the PreToolUse hook. |
| started_at В· ended_at В· last_heartbeat_at | `last_heartbeat_at` is what the reaper watches. |

## events вЂ” "one line in the log" (append-only)

| Column | Notes |
|---|---|
| `seq` | `bigint` identity вЂ” the monotonic key the feed orders by (timestamps can tie). |
| run_id В· project_id В· task_id | Optional FKs, all `SET NULL` вЂ” the log outlives what it references. |
| actor_label В· type В· level | Who acted, what happened (`EVENT_TYPES`), and severity (`debug/info/warn/error`). |
| summary В· payload | Human line plus an optional `jsonb` blob. |
| tokens В· cost_micros | Per-event attribution only вЂ” never the rollup source (that's `runs.cost_micros`). |
| idempotency_key | Unique (partial) вЂ” dedupes retried appends so a re-sent hook can't double-log. |

## composio_toolkits вЂ” "a shared per-toolkit Composio resource cache"

One row per toolkit slug, populated lazily the first time a connection is created. Holds the
shared Composio auth-config and MCP server resources вЂ” never a secret.

| Column | Notes |
|---|---|
| `slug` (PK) | Matches a key in the Composio catalog вЂ” the authority for valid toolkit slugs. |
| auth_config_id | Composio `ac_вЂ¦` auth-config id, created once and reused across projects. |
| mcp_server_id | Composio MCP server id, created once per toolkit. |
| mcp_url | Base MCP URL, e.g. `https://backend.composio.dev/v3/mcp/<id>`. |
| created_at В· updated_at | Timestamps. |

## composio_connections вЂ” "a project's link to one toolkit"

One row per (project, toolkit) pair вЂ” the UNIQUE index enforces the one-connection-per-(project, toolkit) invariant intentionally; see the Indexes section.

| Column | Notes |
|---|---|
| `id` | UUID. |
| project_id | FK to `projects`, `ON DELETE CASCADE` вЂ” delete a project, its connections go too. |
| toolkit_slug | Composio catalog key. Not a FK to `composio_toolkits` вЂ” the cache may not exist yet when a connection is first written. |
| user_id | Composio user identity scoped to this project (`mc-proj-<projectId>`). |
| connected_account_id | Composio `ca_вЂ¦` id, set once the OAuth link is initiated. |
| status | `ConnectionStatus`: `initializing\|active\|error\|expired\|disconnected`. |
| link_url | Transient hosted OAuth link for an in-flight connect; cleared on completion. |
| error | Last error message, if any. |
| created_at В· updated_at | Timestamps. |

## agent_profiles вЂ” "a reusable agent recipe"

Global, slug-addressed capability bundles + auto-routing rules. See [Agent profiles](/profiles)
for the concepts. Churny value sets (`runtime`, `permission_mode`) are validated text, matching
the rest of the schema.

| Column | Notes |
|---|---|
| `id` В· slug В· name | UUID, a unique slug (the address you pass to `mc profile`), and a display name. |
| runtime В· model | `runtime` is `claude-code` or `exec`; `model` is the primary model. |
| fallback_model В· daily_budget_micros | Cost-aware routing: the `--fallback-model` (resilience) and the budget-downgrade target, plus a `bigint` micro-dollar cap on this profile's same-day run cost (`NULL` = no cap). |
| provider В· base_url В· permission_mode | Optional provider/gateway hints and the Claude permission posture. |
| skills В· allowed_tools В· disallowed_tools | `jsonb` string arrays вЂ” steered skills and the tool allow/deny lists. |
| mcp_servers В· env | `jsonb` вЂ” the `--mcp-config` inner map and an env map. Secrets are `${ENV}` placeholders only; the daemon resolves them at spawn. |
| append_system_prompt В· exec_template | Persona text, and the command template required when `runtime='exec'` (tokens `${MODEL}`/`${PROMPT}`/`${MCP_CONFIG}`/`${FALLBACK_MODEL}`). |
| match_rules В· priority В· is_default В· enabled | `jsonb` routing rules (`projectSlugs`/`projectCategories`/`taskKinds`/`labelPattern`), the tie-break, the single global default, and an on/off flag. |
| schedule_enabled В· schedule_project_id | Master on/off for scheduled check-ins, and the FK to the project the check-in runs in (`ON DELETE SET NULL`). |
| schedule_interval_sec В· schedule_cron В· schedule_timezone | The cadence вЂ” an interval in seconds (в‰Ґ60) **or** a cron expression read in an IANA timezone. Cron wins when both are set. |
| check_in_prompt | The standing mission handed to the agent on each scheduled wake-up. |
| last_check_in_at В· consecutive_failures | When the profile last checked in (the "is it due?" baseline, advanced at spawn), and the failure streak that auto-pauses the schedule at three. |
| created_at В· updated_at | Timestamps. |

A **settings** key/value table and the four **BetterAuth** tables (`users`, `sessions`,
`accounts`, `verification`) round out the database. The agent role **cannot see the auth tables
at all** вЂ” see [Access](/access).

## Indexes that matter

| Index | Purpose |
|---|---|
| `tasks_claimable_idx` | Partial over only `status='todo' AND kind='custom'` rows, ordered `(sort_order, created_at)` вЂ” priority by board position, then FIFO. The daemon polls this on a loop, so a board re-order changes what it picks up next. |
| `runs_status_heartbeat_idx` | The reaper's index вЂ” finds `running` runs whose heartbeat went stale. |
| `events_order_idx` | `(created_at, seq)` вЂ” the activity feed's replay order. |
| `events_idempotency_uq` | Partial unique on `idempotency_key` вЂ” the dedupe guarantee. |
| `tasks_project_integration_uq` | One integration row per type per project (keeps the grids from double-counting). |
| `agent_profiles_slug_uq` | Unique profile slug вЂ” the handle you address with `mc profile`. |
| `agent_profiles_default_uq` | Partial unique over `is_default = true` вЂ” enforces *at most one* global default profile. |
| `agent_profiles_schedulable_idx` | Partial over rows with `schedule_enabled` вЂ” the scheduler's scan for profiles that might be due. |
| `composio_connections_project_toolkit_uq` | UNIQUE on `(project_id, toolkit_slug)` вЂ” enforces the intentional one-connection-per-(project, toolkit) invariant. Two connections for the same toolkit in the same project are an explicit non-goal; dropping this index requires a per-connection `user_id` redesign. |

<Note>
**Why two agents never do the same task.** A claim is a **single conditional write**: "set
`claimed_by_run_id` and `claim_expires_at` on this task *only if* it's unclaimed or its claim
has expired." Because it's one indivisible statement, concurrent agents can't both win вЂ” exactly
one row update succeeds; the rest get zero rows back and surface a `CONFLICT`. Claiming is
**orthogonal to status**: it never moves the task to `in_progress`. This is textbook *optimistic
concurrency*, and it's what makes the autopilot safe to run many-at-once.
</Note>
