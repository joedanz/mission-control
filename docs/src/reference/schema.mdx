# The database, column by column

Eleven working tables and the auth tables, defined with Drizzle ORM on Neon Postgres.

Every `id` is a UUID stored as text. Enumerated values are split deliberately into two kinds:
**hard `pgEnum` types** for sets that almost never change, and **plain text validated in the
app/CLI layer** for "churny" sets that do — so adding a status never needs a database migration.

## projects — "a thing you're building"

| Column | Notes |
|---|---|
| `id` · name · `slug` | UUID, display name, and a unique URL-safe handle (the address you pass to `mc`). |
| category · status · accent | `category`/`accent` are pgEnums; `status` is validated text. |
| tech_stack | `jsonb` string array, indexed with a GIN index for tech-based filtering. |
| repo_path · repo_url · live_url | Local checkout (daemon target), GitHub URL (issue import), and the deployed site. |
| priority · target_date · notes | Planning metadata. |
| archived · sort_order | Soft-hide flag and manual ordering. |
| sentry_project_slug | Nullable text. Sentry project slug — when set, enables the Errors tab. |
| email_provider · email_address | Nullable text. Manual provider label and primary email address — when set, enables the Email tab. |
| stripe_site | Nullable text. Stripe `metadata.site` value — when set, enables the Revenue tab. |
| created_at · updated_at · last_activity_at | Timestamps; `last_activity_at` drives the "recently touched" list. |

## tasks — "one item of work"

| Column | Notes |
|---|---|
| `project_id` | FK to `projects`, `ON DELETE CASCADE` — delete a project, its tasks go too. |
| label · notes | The task's title, freeform notes. |
| status | Custom workflow status: `todo→in_progress→done`. |
| version | Optimistic-concurrency token, bumped on every status/claim write — the collision signal. |
| sort_order | Manual board position within a column; lower comes first. Drives the queue head (see `tasks_claimable_idx`) so a board re-order steers what the daemon claims next. |
| claimed_by_run_id | FK to `runs`, `ON DELETE SET NULL` — a deleted/abandoned run frees the claim. |
| claimed_at · claim_expires_at | When the claim was taken and when it auto-releases (the backstop). |
| completed_at | Set when the task reaches `done`. |

## runs — "one AI work-session"

| Column | Notes |
|---|---|
| `id` | Client-supplied by the hook, so a retried `run.start` upserts the same row. |
| agent_label · parent_run_id | Free-text agent name; self-FK builds sub-agent trees. |
| project_id · title · source · model | Context; `source` is `hook\|cli\|cron\|manual` (`cron` = a scheduled check-in). `model` is the model that actually ran (the daemon writes the budget-chosen one). |
| agent_profile_id | FK to `agent_profiles`, `ON DELETE SET NULL` — which profile this run used; deleting the profile leaves the run intact. |
| status | `running → completed \| failed \| abandoned`. "Live/idle" is *derived* from the heartbeat, never stored. |
| tokens_in/out · cache_read/write_tokens · cost_micros | `bigint` cumulative totals (cache split out — it dominates Claude cost); cost in micro-dollars. |
| session_id · work_dir · transcript_ref | Claude Code's session id, the hook's cwd, and a *path* to the transcript — never its body. |
| cancel_requested | The kill-switch flag; surfaced on the heartbeat response and enforced by the PreToolUse hook. |
| started_at · ended_at · last_heartbeat_at | `last_heartbeat_at` is what the reaper watches. |

## events — "one line in the log" (append-only)

| Column | Notes |
|---|---|
| `seq` | `bigint` identity — the monotonic key the feed orders by (timestamps can tie). |
| run_id · project_id · task_id | Optional FKs, all `SET NULL` — the log outlives what it references. |
| actor_label · type · level | Who acted, what happened (`EVENT_TYPES`), and severity (`debug/info/warn/error`). |
| summary · payload | Human line plus an optional `jsonb` blob. |
| tokens · cost_micros | Per-event attribution only — never the rollup source (that's `runs.cost_micros`). |
| idempotency_key | Unique (partial) — dedupes retried appends so a re-sent hook can't double-log. |

## composio_toolkits — "a shared per-toolkit Composio resource cache"

One row per toolkit slug, populated lazily the first time a connection is created. Holds the
shared Composio auth-config and MCP server resources — never a secret.

| Column | Notes |
|---|---|
| `slug` (PK) | Matches a key in the Composio catalog — the authority for valid toolkit slugs. |
| auth_config_id | Composio `ac_…` auth-config id, created once and reused across projects. |
| mcp_server_id | Composio MCP server id, created once per toolkit. |
| mcp_url | Base MCP URL, e.g. `https://backend.composio.dev/v3/mcp/<id>`. |
| created_at · updated_at | Timestamps. |

## mcp_connections — "a project's link to one MCP server"

One row per MCP server attached to a project, discriminated by `source`. The composio-only columns
are nullable (a remote row leaves them null) and the reverse; a DB `CHECK` (`mcp_connections_source_shape`)
enforces that each source carries its required columns. The UNIQUE indexes enforce one Composio
connection per (project, toolkit) and one remote server per (project, name); see the Indexes section.

| Column | Notes |
|---|---|
| `id` | UUID. |
| project_id | FK to `projects`, `ON DELETE CASCADE` — delete a project, its connections go too. |
| source | `composio` (a brokered toolkit, the default) or `remote` (a directly-supplied remote-http server). |
| toolkit_slug | *(composio)* Composio catalog key. Not a FK to `composio_toolkits` — the cache may not exist yet when a connection is first written. Nullable (remote rows leave it null). |
| user_id | *(composio)* Composio user identity scoped to this project (`mc-proj-<projectId>`). Nullable. |
| connected_account_id | *(composio)* Composio `ca_…` id, set once the OAuth link is initiated. Nullable. |
| status | `ConnectionStatus`: `initializing\|active\|error\|expired\|disconnected`. A remote row is pinned `active`. |
| link_url | *(composio)* Transient hosted OAuth link for an in-flight connect; cleared on completion. |
| error | Last error message, if any. |
| remote_name · remote_url · remote_headers | *(remote)* The server's key (unique per project), its URL, and a `jsonb` header map whose values are `${ENV}` placeholders (never a literal secret). Null for composio rows. |
| created_at · updated_at | Timestamps. |

## agent_profiles — "a reusable agent recipe"

Global, slug-addressed capability bundles + auto-routing rules. See [Agent profiles](/profiles)
for the concepts. Churny value sets (`runtime`, `permission_mode`) are validated text, matching
the rest of the schema.

| Column | Notes |
|---|---|
| `id` · slug · name | UUID, a unique slug (the address you pass to `mc profile`), and a display name. |
| runtime · model | `runtime` is `claude-code` or `exec`; `model` is the primary model. |
| fallback_model · daily_budget_micros | Cost-aware routing: the `--fallback-model` (resilience) and the budget-downgrade target, plus a `bigint` micro-dollar cap on this profile's same-day run cost (`NULL` = no cap). |
| provider · base_url · permission_mode | Optional provider/gateway hints and the Claude permission posture. |
| skills · allowed_tools · disallowed_tools | `jsonb` string arrays — declared skills (resolved + enforced at spawn; filesystem or `plugin:skill`) and the tool allow/deny lists. |
| mcp_servers · env | `jsonb` — the `--mcp-config` inner map and an env map. Secrets are `${ENV}` placeholders only; the daemon resolves them at spawn. |
| append_system_prompt · exec_template | Persona text, and the command template required when `runtime='exec'` (tokens `${MODEL}`/`${PROMPT}`/`${MCP_CONFIG}`/`${FALLBACK_MODEL}`). |
| match_rules · priority · is_default · enabled | `jsonb` routing rules (`projectSlugs`/`projectCategories`/`labelPattern`), the tie-break, the single global default, and an on/off flag. |
| schedule_enabled · schedule_project_id | Master on/off for scheduled check-ins, and the FK to the project the check-in runs in (`ON DELETE SET NULL`). |
| schedule_interval_sec · schedule_cron · schedule_timezone | The cadence — an interval in seconds (≥60) **or** a cron expression read in an IANA timezone. Cron wins when both are set. |
| check_in_prompt | The standing mission handed to the agent on each scheduled wake-up. |
| last_check_in_at · consecutive_failures | When the profile last checked in (the "is it due?" baseline, advanced at spawn), and the failure streak that auto-pauses the schedule at three. |
| created_at · updated_at | Timestamps. |

## workflows — "a defined flow of nodes"

One row per [workflow](/workflows) — the authored graph and its lifecycle. Node types, statuses, and
triggers are all validated text (in `lib/workflows.ts`), so a new node type never needs a migration.

| Column | Notes |
|---|---|
| `id` · project_id · slug · name | UUID; the FK to the home `projects` row (`ON DELETE CASCADE`); a **globally unique** slug (the `mc workflow` address); a display name. |
| description | Optional text. |
| status | Validated text: `draft · active · paused`. Only an `active` workflow fires from a cron or event trigger. |
| graph | `jsonb` `{nodes, edges}` — the React Flow authoring graph (each node: `id`, `type`, `position`, `data`). The single source of truth the canvas and CLI both edit. |
| version | Integer, bumped on every graph save — the observability counter. |
| created_at · updated_at | Timestamps; `updated_at` is the cron anchor before a workflow's first scheduled fire. |

## workflow_runs — "one execution of a workflow"

One row per run. Pins a **snapshot** of the graph so editing the workflow can't corrupt a walk in
flight, and carries the payload that fired it.

| Column | Notes |
|---|---|
| `id` · workflow_id | UUID and the FK to `workflows`. |
| status | Validated text: `queued → running → completed \| failed \| cancelled`, plus `paused` (a gate is awaiting a human). `queued` is the daemon's to claim; `paused` is **not** terminal — the reaper skips it, but the single-flight guard still counts it. |
| trigger | How it fired: `manual · cron · event · webhook`. |
| graph_snapshot | `jsonb` — the graph copied at run start; the walker reads this, never the live `workflows.graph`. |
| context | `jsonb` — the trigger payload (e.g. a webhook body), surfaced to the graph as `{{trigger.output.*}}`. |
| cancel_requested | The kill-switch flag; propagated to the in-flight agent node's run. |
| started_at · ended_at · last_heartbeat_at | `last_heartbeat_at` is what the reaper watches to fail a dead `running` walker. |

## workflow_step_runs — "one node's execution within a run"

One row per executed node — idempotent and resumable on the `(workflow_run, node)` unique key.

| Column | Notes |
|---|---|
| `id` · workflow_run_id · node_id | UUID, the FK to `workflow_runs`, and the node's id within the graph. |
| status | Validated text: `pending · running · completed · failed · skipped` (a node off a not-taken branch is `skipped`; a node `running` indefinitely is an awaiting gate). |
| run_id | FK to `runs` (`ON DELETE SET NULL`) — the agent run an **agent** node opened. Null for deterministic nodes (trigger / integration / branch / gate). |
| output | `jsonb` — the full captured result: an agent's result line + `structured_output`, an integration's response `data`, a branch's chosen case, or a gate's decision. The substrate `{{node.field}}` references read from. |
| error | Last error message, if the node failed. |
| started_at · ended_at · created_at | Timestamps. |

## settings — "app-level key/value store"

A tiny key/value table for global app configuration — one row per setting. It's defined and
ready, but no application code reads or writes it yet (it's reserved for future app-level
settings).

| Column | Notes |
|---|---|
| `key` | The setting's name — the primary key, so there's exactly one row per setting. |
| `value` | The setting's value, stored as text. |
| `updated_at` | When the row last changed (`timestamptz`, defaults to now). |

The four **BetterAuth** tables (`users`, `sessions`, `accounts`, `verification`) round out the
database. The agent role **cannot see the auth tables at all** — see [Access](/access).

## Indexes that matter

| Index | Purpose |
|---|---|
| `tasks_claimable_idx` | Partial over only `status='todo'` rows, ordered `(sort_order, created_at)` — priority by board position, then FIFO. The daemon polls this on a loop, so a board re-order changes what it picks up next. |
| `runs_status_heartbeat_idx` | The reaper's index — finds `running` runs whose heartbeat went stale. |
| `events_order_idx` | `(created_at, seq)` — the activity feed's replay order. |
| `events_idempotency_uq` | Partial unique on `idempotency_key` — the dedupe guarantee. |
| `agent_profiles_slug_uq` | Unique profile slug — the handle you address with `mc profile`. |
| `agent_profiles_default_uq` | Partial unique over `is_default = true` — enforces *at most one* global default profile. |
| `agent_profiles_schedulable_idx` | Partial over rows with `schedule_enabled` — the scheduler's scan for profiles that might be due. |
| `mcp_connections_project_toolkit_uq` | UNIQUE on `(project_id, toolkit_slug)` — enforces the intentional one-Composio-connection-per-(project, toolkit) invariant. Two connections for the same toolkit in the same project are an explicit non-goal; dropping this index requires a per-connection `user_id` redesign. |
| `mcp_connections_project_remote_uq` | Partial UNIQUE on `(project_id, remote_name)` `WHERE source='remote'` — one remote server per name per project (composio rows are ignored, since `remote_name` is null). |
| `workflows_slug_uq` | UNIQUE on `slug` — the globally-unique handle you address with `mc workflow`. |
| `workflow_runs_status_heartbeat_idx` | The reaper's index over `workflow_runs` — finds `running` runs whose heartbeat went stale (a dead walker). `queued` and `paused` runs are excluded by design. |
| `workflow_step_runs_run_node_uq` | UNIQUE on `(workflow_run_id, node_id)` — makes step writes idempotent so the walker is resumable: it re-derives completed nodes from the table, not from memory. |

<Note>
**Why two agents never do the same task.** A claim is a **single conditional write**: "set
`claimed_by_run_id` and `claim_expires_at` on this task *only if* it's unclaimed or its claim
has expired." Because it's one indivisible statement, concurrent agents can't both win — exactly
one row update succeeds; the rest get zero rows back and surface a `CONFLICT`. Claiming is
**orthogonal to status**: it never moves the task to `in_progress`. This is textbook *optimistic
concurrency*, and it's what makes the autopilot safe to run many-at-once.
</Note>
