# The `mc` command-line tool

The full surface an agent (or an operator) drives the system with — every command, its output,
and its exit code.

`mc` is a thin layer over the very same `lib/mutations.ts` core the website writes through, so
the CLI and the dashboard **can never drift**. There is no build step — it runs the TypeScript
source directly through `tsx`. Install it once from the repo:

```bash
npm link          # puts `mc` on your PATH
mc --version      # confirm it resolves
```

## Output contract

Always pass `--json` — or just pipe it: JSON is the default whenever stdout is not a terminal,
so agents get machine output automatically. In JSON mode stdout is **exactly one** document;
every log and warning goes to stderr. The envelope is always one of three shapes:

```jsonc
// success — a mutation returns the affected row (so you get the new id)
{ "ok": true,  "command": "project add",  "data": { "id": "…", "slug": "…" } }
// list — items plus a total count (items capped by --limit, default 50)
{ "ok": true,  "command": "project list",  "data": { "items": [ … ], "count": 19 } }
// error — a stable code, a message, and sometimes the offending field
{ "ok": false, "command": "project get", "error": { "code": "NOT_FOUND", "message": "…" } }
```

| Exit code | Meaning | Typical cause |
|---|---|---|
| **0** | OK | Command succeeded. |
| **1** | DB / conflict | A database error, or a lost claim race (`CONFLICT`). |
| **2** | Validation | A bad flag value — the message lists what's allowed. |
| **3** | Not found | No project/task/run by that slug or id. |
| **4** | Config / credentials | Missing or loose-permission credential file. |

## Projects

| Command | What it does |
|---|---|
| `project list [--category --status --archived --search --limit]` | The project register, filterable and searchable. |
| `project get <slug>` | One project with its tasks and MCP connections. |
| `project add --name --category [--status --accent --tech --repo-path --repo-url --live-url --priority --sentry-project --email-provider --email-address --stripe-site …]` | Create a project; returns the new row (and id). |
| `project update <slug> [any add flag]` | Patch — only the flags you pass change. `--sentry-project`, `--email-provider`, `--email-address`, `--stripe-site` are new nullable fields wiring the Errors, Email, and Revenue tabs. |
| `project set-repo <slug> <path> [url]` | Point a project at a local repo (the daemon needs this). |
| `project rm <slug> --yes` | Delete a project; cascades its tasks. Refuses without `--yes`. |

## Tasks

| Command | What it does |
|---|---|
| `task list <slug> [--status]` | A project's tasks. |
| `task add <slug> <label…>` | Add a custom task to the queue. |
| `task set-status <id> <status>` | `todo\|in_progress\|done` — idempotent; prefer over toggle. |
| `task move <id> [--status <s>] [--top \| --after <id>]` | The board move: change a custom task's column (`--status`) and/or reposition it within a column (`--top`, or `--after` a sibling). `--top` puts it next in the claim queue. Refuses a live-claimed card → `CONFLICT`. |
| `task next [--project <slug>]` | Peek the next claimable task (a `custom`, `todo`, unclaimed-or-expired task) — ordered by board position (`sort_order`), then oldest-first. Dragging a card up the board's To Do column promotes it here. |
| `task claim <id> [--run <id>] [--ttl <secs>]` | Atomically take a task for the current run. Loses the race → `CONFLICT` (exit 1). |
| `task import-issues <slug> [--state --label --limit --dry-run]` | GitHub issues → custom tasks. Idempotent by issue number; needs `repoUrl` + the `gh` CLI. |
| `task get` / `toggle` / `rm <id> --yes` | Inspect, flip, or delete a single task. |

## Runs, the event log & spend

| Command | What it does |
|---|---|
| `run start --agent <label> [--project --title --source --model --session-id --work-dir --id]` | Open an agent session; prints the `runId` to capture into `MC_RUN_ID`. |
| `run end <id> <status> [--tokens-in --tokens-out --cache-read --cache-write --cost-micros --authoritative]` | Close a run with cumulative token/cost totals. `--authoritative` **sets** the metrics exactly (the daemon posts claude's own reported cost this way) instead of the default monotonic `GREATEST` guard. |
| `run list [--active --agent --limit]` | Lean rows; what's running right now with `--active`. |
| `run get <id>` | The full run row plus its event trail. |
| `run cancel <id>` | Request cancellation — sets `cancel_requested` (the kill-switch). |
| `event add <summary…> --type [--level --project --task --run --agent]` | Append one line to the activity log. |
| `event list [--project --run --level --limit]` | Newest-first feed at or above a level. |
| `spend [--group-by project\|agent\|day\|run] [--since --until --project --agent --profile]` | Cost rollup — sums `runs.cost_micros`, the authoritative per-run total. `--profile` scopes to one profile (what the daemon's budget check reads). |

## Agent profiles

Named, slug-addressed capability bundles + auto-routing rules. See [Agent profiles](/profiles) for
the concepts; this is the surface. Profiles are global (not per-project).

| Command | What it does |
|---|---|
| `profile list [--enabled --runtime --schedulable]` | All profiles, filterable by enabled state, runtime, or — with `--schedulable` — just the ones with scheduled check-ins armed (the scheduler's scan). |
| `profile get <slug>` | One profile's full config. |
| `profile add --slug --name [--runtime --model --fallback-model --daily-budget-micros --permission-mode --skills --mcp-config --allowed-tools --disallowed-tools --append-system-prompt --env --exec-template --match-project --match-category --match-label --priority --default --disabled --schedule-enabled --schedule-project --schedule-interval --schedule-cron --schedule-timezone --check-in-prompt]` | Create a profile; returns the new row. `runtime=exec` requires `--exec-template`. To enable a schedule: `--schedule-project` plus exactly one of `--schedule-interval` (≥60s) / `--schedule-cron` (read in `--schedule-timezone`). |
| `profile update <slug> [any add flag] [--enabled]` | Patch — only the flags you pass change. Pass `""` to a schedule trigger to clear it. |
| `profile set-default <slug>` | Make this the single global default (clears any other default). Idempotent. |
| `profile checked-in <slug> [--status ok\|fail]` | Record a scheduled check-in's outcome — the scheduler calls this. Bare: advance the check-in clock. `ok` resets the failure counter; `fail` increments it and auto-pauses the schedule after three in a row. |
| `profile resolve [--project --task --label]` | Preview which profile auto-routing would pick — match rules → priority → default — **and** how its declared skills resolve: per skill a `source` (user / project / plugin, with marketplace) or a failure `reason`, plus a `skillsResolved` flag. |
| `profile rm <slug> --yes` | Delete a profile. Runs that used it keep their row (the link just drops). |

<Note>
**Secrets are `${ENV}` placeholders only.** In `--env` values and MCP headers you store a
`${VAR}` placeholder, never the secret. The daemon resolves it from its own environment at spawn
and writes rendered MCP config to a `0600` temp file — so a key never reaches the database or
`ps`. A missing variable fails the run instead of launching half-credentialed. `mc profile add`
warns on stderr if a value *looks* like a raw secret.
</Note>

## Skills

Inspect the skills agents can resolve, and discover + install new ones from the
[skills.sh](https://skills.sh) registry. A profile's `skills` are enforced at spawn; `skill list`
resolves against the same locations the daemon does, so you can preflight a profile before scheduling
it. See [Skills](/profiles#skills) for the concepts.

| Command | What it does |
|---|---|
| `skill list [--project <slug>]` | Every **resolvable** skill, source-tagged: user (`~/.claude/skills`), and the enabled+installed plugin skills (`<plugin>:<skill>`, with marketplace). With `--project`, also that project's work-dir skills and project-enabled plugins. |
| `skill search <query…> [--limit <n>]` | Search the skills.sh registry for **installable** skills. Hits the public, unauthenticated `skills.sh/api/search` (the same endpoint `npx skills` uses — no credentials, no DB). Each result carries an `installed` flag (its slug matched against your local resolvable catalog). |
| `skill add <target> [--force]` | Install a registry skill into `~/.claude/skills/<slug>/`, after which `skill list` and every spawn resolve it. `<target>` is `owner/repo@skill` or a registry id `owner/repo/skill`. Content is fetched from the skill's **public GitHub repo** (no skills.sh auth; an optional `GITHUB_TOKEN` only lifts rate limits). A CLI-local filesystem action — no DB row, no run. Refuses an already-installed skill unless `--force`. |

`skill list` and `skill search` are two distinct surfaces: **resolvable** (what's on disk now) vs.
**installable** (what the registry offers). The catalog and the spawn-time check share one resolver,
so anything `skill list` shows resolves at spawn — and anything it omits will not.

`skill add` is the one place `mc` writes into `~/.claude`. It is safe by construction: the slug is
name-checked, every fetched file path is traversal-guarded, and the `SKILL.md` frontmatter is
validated — all before any write, which then lands atomically. `mc` never hosts or distributes skill
content (it comes from GitHub) and never edits your plugin or settings config. The OIDC-gated
leaderboard/curated browse is not yet exposed.

## MCP servers

Attach MCP servers to a project — by `source`: a Composio-brokered toolkit (OAuth) or a direct
remote-http server. All subcommands emit the standard `{ok,command,data|error}` JSON envelope.
`catalog` has no DB requirement; all others need credentials. See [MCP servers](/mcp) for the
concepts.

| Command | What it does |
|---|---|
| `mcp catalog [--search <q>] [--limit <n>]` | Browse Composio's **full live catalog** (~1,000+ toolkits) — slug, name, tool count, and a `featured` flag for MC's curated set. `--search` is a server-side fuzzy filter; `--limit` caps the page (default 50, max 500). No DB credentials needed. |
| `mcp connect <slug> <toolkit>` | Start a Composio connection to any catalog toolkit; returns the OAuth authorize URL and prints the next step `mc mcp status <slug> <toolkit>`. A curated toolkit binds its narrow tool list; any other binds all of its tools. |
| `mcp status <slug> <toolkit>` | Poll Composio and persist the connection's current status. |
| `mcp list <slug>` | List a project's MCP servers — **both** sources, each tagged `composio` or `remote` — and their statuses. |
| `mcp disconnect <slug> <toolkit>` | Revoke a **Composio** connection at Composio and mark it disconnected locally (composio-only — remote rows use `remove-remote`). |
| `mcp config <slug>` | Resolve a project's active servers (both sources) into one `mcpServers` JSON map — what the daemons use at spawn time. On a key collision a remote server wins over a composio one. |
| `mcp refresh <slug>` | Re-poll all of a project's Composio connections; emit a `composio.connection_changed` event per status transition. |
| `mcp add-remote <slug> --name <n> --url <u> [--header K=V …]` | Attach a `remote` MCP server directly (no OAuth/Composio). `--name` doubles as the server key and is unique per project (idempotent on re-add). Every header value must be an `${ENV}` placeholder — a literal secret is rejected. The row is immediately `active`. |
| `mcp remove-remote <slug> <name>` | Detach a remote MCP server by name (a row delete, no network call). |

<Note>
**Composio tokens expire; remote servers don't.** Composio access tokens can lapse — run
`mc mcp refresh <slug>` periodically (or from a check-in) to re-poll every connection and emit a
`composio.connection_changed` event for any that changed. A `remote` server has no token lifecycle,
so it stays `active` until you `remove-remote` it. See [MCP servers](/mcp) for the full lifecycle.
</Note>

## Workflows

Author and operate [workflows](/workflows) — graphs of nodes (trigger · agent · integration · branch
· gate) executed by the [workflow daemon](/reference/operating#the-workflow-daemon). Graphs are
validated by the same rules the runner uses, so the CLI and canvas can never disagree.

| Command | What it does |
|---|---|
| `workflow list [--project <slug>]` | List workflows (graph size, status, version, latest-run summary). |
| `workflow get <slug>` | One workflow — its graph, status, and version. |
| `workflow create --project <slug> --name <n> [--slug --graph --description]` | Create a workflow from a React Flow `{nodes,edges}` graph (`--graph <json>` or `@file`). A provided graph is validated (one trigger, acyclic, nothing wired into the trigger, agent prompts present, every `{{ref}}` an ancestor); omit `--graph` for an empty draft. Slug defaults to the slugified name (globally unique). |
| `workflow update <slug> [--graph --name --description]` | Replace a workflow's graph (and/or rename) and bump its version — the CLI twin of canvas authoring, validated through the same rules. Omit `--graph` to keep the current graph. |
| `workflow run <slug> [--timeout --max-parallel --allow-concurrent --async]` | Run now. **Synchronous by default** (this process owns the walk; keep prompts short). `--async` enqueues a `queued` run for the workflow daemon and returns. `--max-parallel <n>` caps concurrent nodes (default 4). `--allow-concurrent` bypasses the single-flight guard. |
| `workflow status <runId>` | A run plus its per-node step rows (each agent step links its `runId` and captured output). |
| `workflow cancel <runId>` | Cancel a run. A `queued` or `paused` run is marked `cancelled` outright; a `running` run gets `cancelRequested`, propagated to the in-flight agent node's run. |
| `workflow approve <runId> <nodeId> [--reject --reason <t> --async]` | Decide a **paused gate** and resume. Default approves and re-walks synchronously; `--reject` fails the gate step (its `onError` then applies); `--async` requeues for the daemon. |
| `workflow activate <slug>` | Set status `active` — a cron/event trigger then fires on its schedule / webhook. |
| `workflow pause <slug>` | Set status `paused` (stops cron/event firing). |
| `workflow webhook-url <slug>` | Print the external webhook URL and `X-Hub-Signature-256` signing details for an event-triggered workflow. Read-only. |

<Note>
**Run-only, with a single-flight guard.** An agent node opens a real `runs` row (so cost, heartbeat,
fleet feed, and the kill-switch work) but creates **no** claimable task — so it can't race the
auto-claim daemon. A second run of the same workflow is refused while one is `queued`, `running`, or
`paused` unless you pass `--allow-concurrent`. The graph is snapshotted onto each run, so editing the
workflow never disturbs a walk in flight.
</Note>

| Command | What it does |
|---|---|
| `spec --json` | The whole command catalog, with a `readonly` tag per command. |
| `enums --json` | Every valid value for every enum field. |

<Note>
**How an action gets tied to its agent and run.** State writes and `run`/`event` commands
attribute their audit-log entry to `$MC_AGENT` (default `mc`). Set `MC_AGENT=claude-code` and
`MC_RUN_ID` (the Session hooks write the latter) so an agent's every action is grouped under its
run. Token and cost values are **absolute cumulative** totals — monotonic, so a lower or
out-of-order write never regresses them; cost is integer **micro-dollars**.
</Note>

<Warning>
**Credentials are resolved in order; the file must be locked down.** `mc` reads its connection
string from, in order: **1.** `AGENT_DATABASE_URL` in the environment (best for remote/CI),
**2.** the file named by `$MC_ENV_FILE`, **3.** `~/.config/mc/env` — which **must be
`chmod 600`** or `mc` refuses to run. It will **not** silently fall back to the owner/app
`DATABASE_URL` (opt in with `MC_ALLOW_DATABASE_URL_FALLBACK=1`). Rotate on any suspected leak by
changing the role's password — that invalidates every distributed copy at once:

```sql
ALTER ROLE mc_agent PASSWORD '<new>';
```
</Warning>
