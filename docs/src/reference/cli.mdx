# The `mc` command-line tool

The full surface an agent (or an operator) drives the system with ‚Äî every command, its output,
and its exit code.

`mc` is a thin layer over the very same `lib/mutations.ts` core the website writes through, so
the CLI and the dashboard **can never drift**. There is no build step ‚Äî it runs the TypeScript
source directly through `tsx`. Install it once from the repo:

```bash
npm link          # puts `mc` on your PATH
mc --version      # confirm it resolves
```

## Output contract

Always pass `--json` ‚Äî or just pipe it: JSON is the default whenever stdout is not a terminal,
so agents get machine output automatically. In JSON mode stdout is **exactly one** document;
every log and warning goes to stderr. The envelope is always one of three shapes:

```jsonc
// success ‚Äî a mutation returns the affected row (so you get the new id)
{ "ok": true,  "command": "project add",  "data": { "id": "‚Ä¶", "slug": "‚Ä¶" } }
// list ‚Äî items plus a total count (items capped by --limit, default 50)
{ "ok": true,  "command": "project list",  "data": { "items": [ ‚Ä¶ ], "count": 19 } }
// error ‚Äî a stable code, a message, and sometimes the offending field
{ "ok": false, "command": "project get", "error": { "code": "NOT_FOUND", "message": "‚Ä¶" } }
```

| Exit code | Meaning | Typical cause |
|---|---|---|
| **0** | OK | Command succeeded. |
| **1** | DB / conflict | A database error, or a lost claim race (`CONFLICT`). |
| **2** | Validation | A bad flag value ‚Äî the message lists what's allowed. |
| **3** | Not found | No project/task/run by that slug or id. |
| **4** | Config / credentials | Missing or loose-permission credential file. |

## Projects

| Command | What it does |
|---|---|
| `project list [--category --status --archived --search --limit]` | The project register, filterable and searchable. |
| `project get <slug>` | One project with its tasks and integrations. |
| `project add --name --category [--status --accent --tech --repo-path --repo-url --live-url --priority --sentry-project --email-provider --email-address --stripe-site ‚Ä¶]` | Create a project; returns the new row (and id). |
| `project update <slug> [any add flag]` | Patch ‚Äî only the flags you pass change. `--sentry-project`, `--email-provider`, `--email-address`, `--stripe-site` are new nullable fields wiring the Errors, Email, and Revenue tabs. |
| `project set-repo <slug> <path> [url]` | Point a project at a local repo (the daemon needs this). |
| `project rm <slug> --yes` | Delete a project; cascades its tasks. Refuses without `--yes`. |

## Tasks

| Command | What it does |
|---|---|
| `task list <slug> [--status --kind]` | A project's tasks (custom or integration). |
| `task add <slug> <label‚Ä¶>` | Add a custom task to the queue. |
| `task set-status <id> <status>` | `todo\|in_progress\|done` ‚Äî idempotent; prefer over toggle. |
| `task move <id> [--status <s>] [--top \| --after <id>]` | The board move: change a custom task's column (`--status`) and/or reposition it within a column (`--top`, or `--after` a sibling). `--top` puts it next in the claim queue. Refuses a live-claimed card ‚Üí `CONFLICT`. |
| `task next [--project <slug>]` | Peek the next claimable task (a `custom`, `todo`, unclaimed-or-expired task) ‚Äî ordered by board position (`sort_order`), then oldest-first. Dragging a card up the board's To Do column promotes it here. |
| `task claim <id> [--run <id>] [--ttl <secs>]` | Atomically take a task for the current run. Loses the race ‚Üí `CONFLICT` (exit 1). |
| `task import-issues <slug> [--state --label --limit --dry-run]` | GitHub issues ‚Üí custom tasks. Idempotent by issue number; needs `repoUrl` + the `gh` CLI. |
| `task get` / `toggle` / `rm <id> --yes` | Inspect, flip, or delete a single task. |

## Runs, the event log & spend

| Command | What it does |
|---|---|
| `run start --agent <label> [--project --title --source --model --session-id --work-dir --id]` | Open an agent session; prints the `runId` to capture into `MC_RUN_ID`. |
| `run end <id> <status> [--tokens-in --tokens-out --cache-read --cache-write --cost-micros --authoritative]` | Close a run with cumulative token/cost totals. `--authoritative` **sets** the metrics exactly (the daemon posts claude's own reported cost this way) instead of the default monotonic `GREATEST` guard. |
| `run list [--active --agent --limit]` | Lean rows; what's running right now with `--active`. |
| `run get <id>` | The full run row plus its event trail. |
| `run cancel <id>` | Request cancellation ‚Äî sets `cancel_requested` (the kill-switch). |
| `event add <summary‚Ä¶> --type [--level --project --task --run --agent]` | Append one line to the activity log. |
| `event list [--project --run --level --limit]` | Newest-first feed at or above a level. |
| `spend [--group-by project\|agent\|day\|run] [--since --until --project --agent --profile]` | Cost rollup ‚Äî sums `runs.cost_micros`, the authoritative per-run total. `--profile` scopes to one profile (what the daemon's budget check reads). |

## Agent profiles

Named, slug-addressed capability bundles + auto-routing rules. See [Agent profiles](/profiles) for
the concepts; this is the surface. Profiles are global (not per-project).

| Command | What it does |
|---|---|
| `profile list [--enabled --runtime --schedulable]` | All profiles, filterable by enabled state, runtime, or ‚Äî with `--schedulable` ‚Äî just the ones with scheduled check-ins armed (the scheduler's scan). |
| `profile get <slug>` | One profile's full config. |
| `profile add --slug --name [--runtime --model --fallback-model --daily-budget-micros --permission-mode --skills --mcp-config --allowed-tools --disallowed-tools --append-system-prompt --env --exec-template --match-project --match-category --match-kind --match-label --priority --default --disabled --schedule-enabled --schedule-project --schedule-interval --schedule-cron --schedule-timezone --check-in-prompt]` | Create a profile; returns the new row. `runtime=exec` requires `--exec-template`. To enable a schedule: `--schedule-project` plus exactly one of `--schedule-interval` (‚â•60s) / `--schedule-cron` (read in `--schedule-timezone`). |
| `profile update <slug> [any add flag] [--enabled]` | Patch ‚Äî only the flags you pass change. Pass `""` to a schedule trigger to clear it. |
| `profile set-default <slug>` | Make this the single global default (clears any other default). Idempotent. |
| `profile checked-in <slug> [--status ok\|fail]` | Record a scheduled check-in's outcome ‚Äî the scheduler calls this. Bare: advance the check-in clock. `ok` resets the failure counter; `fail` increments it and auto-pauses the schedule after three in a row. |
| `profile resolve [--project --task --label --kind]` | Preview which profile auto-routing would pick ‚Äî match rules ‚Üí priority ‚Üí default. |
| `profile rm <slug> --yes` | Delete a profile. Runs that used it keep their row (the link just drops). |

<Note>
**Secrets are `${ENV}` placeholders only.** In `--env` values and MCP headers you store a
`${VAR}` placeholder, never the secret. The daemon resolves it from its own environment at spawn
and writes rendered MCP config to a `0600` temp file ‚Äî so a key never reaches the database or
`ps`. A missing variable fails the run instead of launching half-credentialed. `mc profile add`
warns on stderr if a value *looks* like a raw secret.
</Note>

## Composio integrations

Manage per-project connections to third-party toolkits via Composio. All subcommands emit the
standard `{ok,command,data|error}` JSON envelope. `catalog` has no DB requirement; all others
need credentials.

| Command | What it does |
|---|---|
| `composio catalog` | List all supported toolkits ‚Äî slug, name, and tool count. No DB credentials needed. |
| `composio connect <slug> <toolkit>` | Start a connection; returns the OAuth authorize URL and prints the next step `mc composio status <slug> <toolkit>`. |
| `composio status <slug> <toolkit>` | Poll Composio and persist the connection's current status. |
| `composio list <slug>` | List a project's connections and their statuses. |
| `composio disconnect <slug> <toolkit>` | Revoke the connection at Composio and mark it disconnected locally. |
| `composio mcp-config <slug>` | Resolve a project's active connections into an `mcpServers` JSON map ‚Äî what the daemons use at spawn time. |
| `composio refresh <slug>` | Re-poll all of a project's connections; emit a `composio.connection_changed` event per status transition. |

<Note>
**Tokens expire.** Composio access tokens can expire. Run `mc composio refresh <slug>` periodically
(or on a schedule) to keep a project's connections live ‚Äî each call re-polls every connection and
emits a `composio.connection_changed` event for any that changed. See [Integrations](/integrations)
for the full connection lifecycle.
</Note>

## Integrations & self-description

| Command | What it does |
|---|---|
| `integration set <slug> <type> <status>` | Upsert by (project, type); idempotent. |
| `integration list <slug>` | A project's plumbing status. |
| `spec --json` | The whole command catalog, with a `readonly` tag per command. |
| `enums --json` | Every valid value for every enum field. |

<Note>
**How an action gets tied to its agent and run.** State writes and `run`/`event` commands
attribute their audit-log entry to `$MC_AGENT` (default `mc`). Set `MC_AGENT=claude-code` and
`MC_RUN_ID` (the Session hooks write the latter) so an agent's every action is grouped under its
run. Token and cost values are **absolute cumulative** totals ‚Äî monotonic, so a lower or
out-of-order write never regresses them; cost is integer **micro-dollars**.
</Note>

<Warning>
**Credentials are resolved in order; the file must be locked down.** `mc` reads its connection
string from, in order: **1.** `AGENT_DATABASE_URL` in the environment (best for remote/CI),
**2.** the file named by `$MC_ENV_FILE`, **3.** `~/.config/mc/env` ‚Äî which **must be
`chmod 600`** or `mc` refuses to run. It will **not** silently fall back to the owner/app
`DATABASE_URL` (opt in with `MC_ALLOW_DATABASE_URL_FALLBACK=1`). Rotate on any suspected leak by
changing the role's password ‚Äî that invalidates every distributed copy at once:

```sql
ALTER ROLE mc_agent PASSWORD '<new>';
```
</Warning>
